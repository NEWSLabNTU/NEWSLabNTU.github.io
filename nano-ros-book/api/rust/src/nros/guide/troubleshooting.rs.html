1//! # Troubleshooting
2//!
3//! ## Message too large / truncated
4//!
5//! Messages pass through multiple buffer layers.  A message must fit every
6//! layer to be delivered intact:
7//!
8//! | Layer | Env var | Posix default |
9//! |-------|---------|---------------|
10//! | Defragmentation | `ZPICO_FRAG_MAX_SIZE` | 65536 |
11//! | Batch size | `ZPICO_BATCH_UNICAST_SIZE` | 65536 |
12//! | Shim buffer | `ZPICO_SUBSCRIBER_BUFFER_SIZE` | 1024 |
13//! | User receive buffer | — (const generic) | 1024 |
14//!
15//! For large messages, increase the transport limits (set before `cargo
16//! build`) and use `_sized` method variants for larger per-entity buffers:
17//!
18//! ```ignore
19//! // 4 KB receive buffer
20//! executor.register_subscription_sized::<MyMsg, _, 4096>("/topic", |msg| { ... })?;
21//! ```
22//!
23//! ## zenoh version mismatch
24//!
25//! zenoh-pico and zenohd must be the same version (1.6.2).  Symptoms:
26//! `z_publisher_put failed: -100` (`_Z_ERR_TRANSPORT_TX_FAILED`) followed
27//! by `-73` (`_Z_ERR_SESSION_CLOSED`).
28//!
29//! Build zenohd from the pinned submodule (`just build-zenohd`) or install
30//! the matching version.
31//!
32//! ## Build issues
33//!
34//! - **Submodule not found** — run `git submodule update --init --recursive`
35//! - **CMake cache stale** (changed env vars not taking effect) — run
36//!   `cargo clean -p zpico-sys` then rebuild
37//!
38//! ## zenoh-pico error codes
39//!
40//! | Code | Name | Meaning |
41//! |------|------|---------|
42//! | -3 | `_Z_ERR_TRANSPORT_OPEN_FAILED` | Cannot connect to router |
43//! | -73 | `_Z_ERR_SESSION_CLOSED` | Session closed after failure |
44//! | -78 | `_Z_ERR_SYSTEM_OUT_OF_MEMORY` | Allocation failed |
45//! | -100 | `_Z_ERR_TRANSPORT_TX_FAILED` | Transport transmission failed |
46//! | -128 | `_Z_ERR_GENERIC` | Generic error |
